// What Is This & Why Does It Matter
MITRE ATT&CK — The Rosetta Stone of Cybersecurity
Imagine every security team worldwide used different names for the same attack technique — impossible to share intelligence or compare tools. MITRE ATT&CK solved that problem for cybersecurity. It's a standardised catalogue of every known attack technique used by real hackers against real organisations — organised in a way that defenders, attackers, and tools can all speak the same language.

Every technique has a unique ID (like T1110.003), a description of what the attacker does, what logs you'd see if it happened, and how to stop it. When a security engineer says "we detected T1078" — everyone in the room immediately knows what that means, what happened, and what to do.

You don't need to memorise the entire matrix. You need to understand the structure, know how to look techniques up, and get comfortable tagging your work with technique IDs. The rest comes with practice.
Real-world analogy: Think of ATT&CK like a universal parts catalog for mechanics. A mechanic doesn't memorize every part number — but they know the system, know how to look up a part, and use the catalog to communicate precisely with suppliers and colleagues. ATT&CK works the same way for security teams worldwide.

💡 Plain English — What This Actually Means

MITRE ATT&CK is the master reference guide for how attackers operate. Think of it like the operations manual that every thief uses — and that every security guard studies. MITRE (a US non-profit funded by the government) analyzed thousands of real cyberattacks and documented every method attackers used. They organized it into a matrix: rows are the attacker's goals, cells are the specific methods to achieve them. Once you know the playbook, you know exactly where to set your tripwires.

Understanding the Tactic → Technique → Sub-technique hierarchy. A Tactic is the goal. A Technique is the method. A Sub-technique is the specific variant. T1110 is Brute Force. T1110.003 is Password Spraying. You write detections at the sub-technique level.

// What is this phase about?
A SIEM (Security Information and Event Management) is the central nervous system of a security team. Every device, application, and service in an organization generates logs — records of what happened and when. On their own those logs are useless because there are millions of them. A SIEM collects them all in one place, lets you search them, and automatically fires an alert when it detects a suspicious pattern.

Microsoft Sentinel is Microsoft's cloud-based SIEM. It stores logs in a database called a Log Analytics Workspace. You search those logs using a query language called KQL (Kusto Query Language). When a query finds something suspicious it creates an Incident — which is what a security analyst opens and investigates.

Think of it this way: Think of Sentinel as a security operations center with cameras everywhere. Every camera (data connector) feeds footage (logs) into a central recording system (Log Analytics). Automated motion detection rules (analytics rules) watch the feeds and trigger an alarm (incident) when something suspicious appears. You are building this entire system from scratch in this phase.
💡 KQL vs SQL: If you have ever written a database query, KQL feels familiar. Instead of SELECT * FROM table WHERE condition, you write table | where condition | summarize count(). Same idea, different syntax. You do not need to memorize KQL — you will learn it by writing real queries in the labs.
// What Is This & Why Does It Matter
Phase 1 — Microsoft Sentinel & KQL: Your First SIEM
A SIEM (Security Information and Event Management) is the central nervous system of a security team. Every device, application, and service in an organisation generates logs — records of what happened and when. A SIEM collects all of those logs in one place, lets you search through them, and automatically alerts you when something suspicious happens.

Microsoft Sentinel is Microsoft's cloud-based SIEM. It sits on top of a database called a Log Analytics Workspace — think of it as a giant spreadsheet where every row is a log entry (a login, a file access, a firewall block). You search that spreadsheet using a query language called KQL.

KQL (Kusto Query Language) is how you ask questions of your log data. "Show me every failed login in the last hour." "Which IP addresses tried to log in to 10 different accounts?" "Did anyone access a file at 3am?" KQL is the tool that turns raw logs into answers.

You don't need to be a programmer to learn KQL. If you can write an Excel formula or understand the concept of filtering a spreadsheet, you can learn KQL. The syntax is different but the logic is the same: filter rows, group results, count things, find patterns.
Real-world analogy: A SIEM is like a city's 911 dispatch center — it receives calls (logs) from every police, fire, and ambulance unit (data sources) in one place. KQL is the query you run when you want to see "every call from downtown in the last 6 hours." The logs are the call records. The analytics rules are the conditions that trigger a dispatch. The incident is the case that gets assigned to a responder.

Used KQL to correlate sign-in logs against a watchlist of known bad IPs, reducing triage time from 20 min to 3 min.

// ENV: Microsoft Sentinel · Log Analytics
Sentinel → Watchlists → Create → upload a CSV with 3 columns: IPAddress, ThreatType, Source. Add 5 fake IPs including one that exists in your SigninLogs.
Write this KQL: let badIPs = _GetWatchlist('ThreatIPs') | project IPAddress; SigninLogs | where IPAddress in (badIPs) | project TimeGenerated, UserPrincipalName, IPAddress, Location

💡 Plain English — What This Actually Means

A watchlist is a list of things you already know are bad — like a bank's list of flagged account numbers or a club's banned customer list. When a login comes from an IP address on your known-bad list, you want to know immediately. Correlating means joining two data sources together: the live login events AND your watchlist. Instead of manually reviewing 10,000 logins, Sentinel automatically surfaces the 3 that came from known bad IPs. KQL (Kusto Query Language) is how you write these questions — conceptually the same as SQL which you already know from your DBA work.

Built a password spray detection rule — grouped failed logins by src IP across 10+ accounts within 5 minutes, tuned to cut false positives by 60%.
▶ LAB
💡 EXPLAIN
// ENV: Microsoft Sentinel · SigninLogs
Open Sentinel → Logs → paste this KQL: SigninLogs | where ResultType != 0 | where TimeGenerated > ago(5m) | summarize FailedAttempts=count(), DistinctUsers=dcount(UserPrincipalName) by IPAddress | where DistinctUsers >= 5
Run it. Adjust the DistinctUsers threshold (5 for lab, 10 for production) until you get meaningful results without false positives from your own test logins.
Promote to an Analytics Rule: Scheduled, runs every 5 min, lookback 5 min. Alert threshold: results > 0.
Simulate the attack: in your M365 dev tenant, make 6 failed login attempts from different user accounts using wrong passwords. Verify the alert fires within 10 minutes.
💡 Plain English — What This Actually Means

Password spraying is a specific hacking technique. Instead of trying 1,000 passwords against one account (which triggers lockout after 5 failed attempts), attackers try 1 very common password — like Summer2024! — against 1,000 different accounts. None lock out, but the attacker finds every account that uses that common password. The detection logic: if the same IP address fails to log in on 5 or more different accounts within 5 minutes — that pattern isn't a user forgetting their password, it's a spray. Tuning the threshold means adjusting that number to balance catching real attacks vs false alarms from IT automation tools.

Wrote time-series anomaly queries using series_decompose_anomalies() to baseline normal login volume and alert only on significant spikes.
▶ LAB
💡 EXPLAIN
// ENV: Microsoft Sentinel · Log Analytics
Run this KQL: SigninLogs | where TimeGenerated > ago(14d) | summarize LoginCount=count() by bin(TimeGenerated, 1h) | make-series Logins=sum(LoginCount) on TimeGenerated step 1h | extend anomalies=series_decompose_anomalies(Logins, 1.5)
Render as a timechart: add | mv-expand TimeGenerated, Logins, anomalies | where anomalies != 0 to surface only anomalous hours.
Note which hours return anomalies=1 (spike) or anomalies=-1 (drop). These are your detection candidates.
Wrap in an Analytics Rule. This rule requires 14 days of data to baseline — use it after your M365 tenant has been active for 2+ weeks.
💡 Plain English — What This Actually Means

Think of a car's traction control system — it doesn't alert every time you accelerate, only when wheel spin exceeds what's statistically normal for your driving conditions. series_decompose_anomalies() works the same way on login data. First it learns your normal login volume day by day, hour by hour. Then it watches for spikes that are statistically unlikely given that baseline. Monday morning login surge after a weekend = normal. 3am Sunday spike = anomalous. The algorithm knows the difference automatically — far smarter than a simple 'alert if more than 100 logins' rule.

Used let statements and functions to modularize a 200-line threat hunt query into reusable components shared across the team.
▶ LAB
💡 EXPLAIN
// ENV: Microsoft Sentinel · Log Analytics
Write a base detection query (e.g. failed logins). Then refactor it: extract the time window into a let: let lookback = 24h;
Extract the threshold: let failThreshold = 10;
Wrap the core logic in a function: .create-or-alter function FailedLoginsByIP() { SigninLogs | where ResultType != 0 | where TimeGenerated > ago(lookback) | summarize count() by IPAddress | where count_ > failThreshold }
Save the function in Sentinel → Logs → Functions. Now any analyst can call FailedLoginsByIP() in any query — one-line invocation instead of rewriting the logic.
💡 Plain English — What This Actually Means

Same concept as SQL stored procedures from your DBA work — write the logic once, give it a name, call it everywhere. Instead of writing the same 50-line detection query in 10 different rules, you write it once as a KQL function and reference it by name. When the logic needs to change (new threshold, new exclusion), you update it in one place and all 10 rules update automatically. Without functions, changing a detection threshold means manually updating every rule that uses it — and risking missing one.

Phase 3 — Azure Network Security: Building Walls, Gates & Checkpoints
Before you can secure a network, you need to understand how it's built. In Azure, everything lives inside Virtual Networks (VNets) — think of them as private neighbourhoods in the cloud. Resources inside a VNet can talk to each other. Resources outside cannot — unless you explicitly allow it.

// What is this phase about?
Identity is the new perimeter. In the old days, security meant building a wall around your network — if you were inside the wall, you were trusted. Today, people work from home, from phones, from coffee shops — the wall is gone. The only thing that proves you are who you say you are is your identity: your username, password, and verification method.

IAM (Identity and Access Management) is the system that controls who can log in, what they can access, and for how long. Instead of someone having admin rights 24/7 — which is dangerous — PIM makes admin access time-limited and requires a justification to activate it. Like a contractor who only gets a master key for the specific hours they're working on site — not a permanent copy they carry forever.

SOAR (Security Orchestration, Automation and Response) is about automating the repetitive parts of incident response. Instead of an analyst manually checking every alert, a SOAR playbook does it automatically — checking user history, creating tickets, sending notifications — so humans only deal with things that truly need human judgment.
💡 Think of IAM like a corporate office's badge access system. Not everyone can enter every room. An intern can access the open workspace but not the server room. A senior engineer can access both but needs to badge in each time (MFA). The data center requires manager approval and a time-limited access code (PIM). Every door opened is logged (audit logs). You are building and securing that access system.
// What Is This & Why Does It Matter
Phase 4 — IAM & Identity Security: Who Are You and What Can You Do?
Identity is the new perimeter. In the old days of security, you protected the network edge — if you were inside the building you were trusted. That model is dead. Today, attackers get inside (via phishing, credential theft, stolen tokens) and then move laterally using legitimate-looking identities. The attacker isn't breaking down the door — they're walking in with a stolen badge.

IAM (Identity and Access Management) is the discipline of controlling who can access what, with how much permission, for how long. The guiding principle is Least Privilege — every person and every application gets the minimum access needed to do their job, nothing more.

Conditional Access is how you enforce that. Instead of just asking "do you have a password?" it asks: "Are you who you say you are? Are you on a trusted device? Are you logging in from a normal location? Is this a normal time?" If anything looks wrong — MFA is required, or access is blocked entirely.

PIM (Privileged Identity Management) solves the problem of standing access — admin accounts that are always powerful, 24/7, even when nobody is using them. With PIM, admin access is off by default and turned on only when needed, for a limited time, with a business justification logged. 💡 Plain English — What This Actually Means

Traditional SIEM alerting: 200 failed logins generate 200 separate alerts. Alert fatigue is when analysts are overwhelmed by the volume and start missing the real threats buried in the noise. Risk-based alerting fixes this by treating each event as a contributor to a risk score rather than a standalone alert. One failed login adds 5 points. A PowerShell execution adds 30 points. A credential dump attempt adds 100 points. Only when a user's or machine's cumulative score crosses a threshold does an investigation get triggered. The result: one investigation into a user with 6 suspicious behaviors — far more actionable than 6 separate low-priority alerts.

💡 Plain English — What This Actually Means

Palo Alto NGFWs use a zone-based model — traffic can only flow between zones if there's an explicit policy permitting it, and everything else is denied by default. Trust zone = internal network. Untrust zone = internet. DMZ = public-facing servers. App-ID is Palo Alto's application identification capability: instead of allowing 'port 443' (which any application including attacker C2 can use), you allow 'the web-browsing application' specifically. App-ID identifies applications by their behavior, not their port — so attacker command-and-control traffic disguised as HTTPS on port 443 can still be identified and blocked because it doesn't behave like normal web browsing.
