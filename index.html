// What Is This & Why Does It Matter
MITRE ATT&CK — The Rosetta Stone of Cybersecurity
Imagine you're a doctor and every hospital in the world uses a different name for the same disease. Chaos. MITRE ATT&CK solved that problem for cybersecurity. It's a standardised catalogue of every known attack technique used by real hackers against real organisations — organised in a way that defenders, attackers, and tools can all speak the same language.

Every technique has a unique ID (like T1110.003), a description of what the attacker does, what logs you'd see if it happened, and how to stop it. When a security engineer says "we detected T1078" — everyone in the room immediately knows what that means, what happened, and what to do.

You don't need to memorise the entire matrix. You need to understand the structure, know how to look techniques up, and get comfortable tagging your work with technique IDs. The rest comes with practice.
Healthcare analogy: Think of ATT&CK like ICD-10 codes in healthcare. A nurse doesn't memorise every ICD-10 code — but she knows the system, knows how to look up a code, and uses them to communicate precisely with the rest of the care team. ATT&CK works the same way for security teams.

// What is this phase about?
A SIEM (Security Information and Event Management) is the central nervous system of a security team. Every device, application, and service in an organization generates logs — records of what happened and when. On their own those logs are useless because there are millions of them. A SIEM collects them all in one place, lets you search them, and automatically fires an alert when it detects a suspicious pattern.

Microsoft Sentinel is Microsoft's cloud-based SIEM. It stores logs in a database called a Log Analytics Workspace. You search those logs using a query language called KQL (Kusto Query Language). When a query finds something suspicious it creates an Incident — which is what a security analyst opens and investigates.

Think of it this way: If your organization is a hospital, Sentinel is the central patient monitoring system. Every patient (device/user) has sensors (data connectors) sending readings (logs) to the central monitor (Log Analytics). Doctors (analytics rules) watch the readings and trigger an alarm (incident) when something looks wrong. You are building the monitoring system from scratch in this phase.
💡 KQL vs SQL: If you have ever written a database query, KQL feels familiar. Instead of SELECT * FROM table WHERE condition, you write table | where condition | summarize count(). Same idea, different syntax. You do not need to memorize KQL — you will learn it by writing real queries in the labs.
// What Is This & Why Does It Matter
Phase 1 — Microsoft Sentinel & KQL: Your First SIEM
A SIEM (Security Information and Event Management) is the central nervous system of a security team. Every device, application, and service in an organisation generates logs — records of what happened and when. A SIEM collects all of those logs in one place, lets you search through them, and automatically alerts you when something suspicious happens.

Microsoft Sentinel is Microsoft's cloud-based SIEM. It sits on top of a database called a Log Analytics Workspace — think of it as a giant spreadsheet where every row is a log entry (a login, a file access, a firewall block). You search that spreadsheet using a query language called KQL.

KQL (Kusto Query Language) is how you ask questions of your log data. "Show me every failed login in the last hour." "Which IP addresses tried to log in to 10 different accounts?" "Did anyone access a file at 3am?" KQL is the tool that turns raw logs into answers.

You don't need to be a programmer to learn KQL. If you can write an Excel formula or understand the concept of filtering a spreadsheet, you can learn KQL. The syntax is different but the logic is the same: filter rows, group results, count things, find patterns.
Healthcare analogy: A SIEM is like a hospital's patient monitoring system — it pulls data from every monitor (heart rate, blood pressure, oxygen) into one central display. KQL is like the query you run when you want to see "every patient whose oxygen dropped below 90% in the last 6 hours." The logs are the vitals. The analytics rules are the alarm thresholds. The incident is the alert that fires when a patient needs attention.

// KQL Detection Writing
Used KQL to correlate sign-in logs against a watchlist of known bad IPs, reducing triage time from 20 min to 3 min.
▶ LAB
💡 EXPLAIN
// ENV: Microsoft Sentinel · Log Analytics
Sentinel → Watchlists → Create → upload a CSV with 3 columns: IPAddress, ThreatType, Source. Add 5 fake IPs including one that exists in your SigninLogs.
Write this KQL: let badIPs = _GetWatchlist('ThreatIPs') | project IPAddress; SigninLogs | where IPAddress in (badIPs) | project TimeGenerated, UserPrincipalName, IPAddress, Location
Run it. If it returns rows — you have a working IOC-based detection. If not, add a real IP from your SigninLogs to the watchlist CSV and re-upload.
Save as a new Analytics Rule: set schedule to run every 5 minutes, entity mapping to Account (UserPrincipalName) and IP (IPAddress).
💡 Plain English — What This Actually Means

A watchlist is a list of things you already know are bad — like a hospital's list of known fraud patterns. Correlating means joining two data sources: live login events AND the watchlist. Instead of manually reviewing 10,000 logins, Sentinel automatically flags the 3 that came from known bad IPs. KQL (Kusto Query Language) is the language you use to ask these questions — conceptually identical to SQL but with different syntax.

Built a password spray detection rule — grouped failed logins by src IP across 10+ accounts within 5 minutes, tuned to cut false positives by 60%.
▶ LAB
💡 EXPLAIN
// ENV: Microsoft Sentinel · SigninLogs
Open Sentinel → Logs → paste this KQL: SigninLogs | where ResultType != 0 | where TimeGenerated > ago(5m) | summarize FailedAttempts=count(), DistinctUsers=dcount(UserPrincipalName) by IPAddress | where DistinctUsers >= 5
Run it. Adjust the DistinctUsers threshold (5 for lab, 10 for production) until you get meaningful results without false positives from your own test logins.
Promote to an Analytics Rule: Scheduled, runs every 5 min, lookback 5 min. Alert threshold: results > 0.
Simulate the attack: in your M365 dev tenant, make 6 failed login attempts from different user accounts using wrong passwords. Verify the alert fires within 10 minutes.
💡 Plain English — What This Actually Means

Password spraying is a specific hacking technique: instead of trying 1,000 passwords against one account (which triggers lockout), attackers try 1 common password against 1,000 accounts. None lock out, but the attacker finds accounts with weak passwords. The detection: if the same IP has failed logins on 5+ different accounts within 5 minutes — that's a spray, not a user forgetting their password. Tuning means adjusting the threshold to balance catching real attacks vs false alarms.

Wrote time-series anomaly queries using series_decompose_anomalies() to baseline normal login volume and alert only on significant spikes.
▶ LAB
💡 EXPLAIN
// ENV: Microsoft Sentinel · Log Analytics
Run this KQL: SigninLogs | where TimeGenerated > ago(14d) | summarize LoginCount=count() by bin(TimeGenerated, 1h) | make-series Logins=sum(LoginCount) on TimeGenerated step 1h | extend anomalies=series_decompose_anomalies(Logins, 1.5)
Render as a timechart: add | mv-expand TimeGenerated, Logins, anomalies | where anomalies != 0 to surface only anomalous hours.
Note which hours return anomalies=1 (spike) or anomalies=-1 (drop). These are your detection candidates.
Wrap in an Analytics Rule. This rule requires 14 days of data to baseline — use it after your M365 tenant has been active for 2+ weeks.
💡 Plain English — What This Actually Means

Like a hospital heart monitor that alerts on abnormal rate compared to baseline (not every time your heart beats faster), this query learns your normal login volume and only alerts on statistically unlikely spikes. Monday morning login surge = normal. 3am Sunday spike = anomalous. The algorithm knows the difference automatically — smarter than a simple threshold rule.

Used let statements and functions to modularize a 200-line threat hunt query into reusable components shared across the team.
▶ LAB
💡 EXPLAIN
// ENV: Microsoft Sentinel · Log Analytics
Write a base detection query (e.g. failed logins). Then refactor it: extract the time window into a let: let lookback = 24h;
Extract the threshold: let failThreshold = 10;
Wrap the core logic in a function: .create-or-alter function FailedLoginsByIP() { SigninLogs | where ResultType != 0 | where TimeGenerated > ago(lookback) | summarize count() by IPAddress | where count_ > failThreshold }
Save the function in Sentinel → Logs → Functions. Now any analyst can call FailedLoginsByIP() in any query — one-line invocation instead of rewriting the logic.
💡 Plain English — What This Actually Means

Same concept as SQL stored procedures — write the logic once, call it by name everywhere. When the detection logic needs to change, you update it in one place and all rules using that function update automatically. Without functions, changing a threshold means manually updating 10 rules and risking missing one. With your DBA background, this maps directly to how you already think about reusable code.

// What is this phase about?
If Sentinel is the monitoring system, Microsoft Defender is the immune system. While Sentinel watches logs after the fact, Defender sits directly on endpoints (computers), identities (user accounts), email, and cloud storage — and actively blocks or alerts on threats in real time.

Defender for Endpoint (MDE) is an agent installed on Windows machines that watches every process, file, registry change, and network connection. When a hacker runs a malicious script, MDE sees it at the process level — not just in a log file minutes later.

Defender for Identity (MDI) watches your Active Directory — the system that manages user accounts and permissions. It detects when attackers try to steal credentials, move between machines, or impersonate other users.

Defender for Storage scans files uploaded to Azure blob storage for malware — like antivirus but for cloud storage buckets.
💡 Think of MDE as a security camera installed directly on each computer — it records everything happening on that machine at the process level. Sentinel is the security operations center watching all the camera feeds. // What is this phase about?

Phase 3 — Azure Network Security: Building Walls, Gates & Checkpoints
Before you can secure a network, you need to understand how it's built. In Azure, everything lives inside Virtual Networks (VNets) — think of them as private neighbourhoods in the cloud. Resources inside a VNet can talk to each other. Resources outside cannot — unless you explicitly allow it.

Phase 4 — IAM & Identity Security: Who Are You and What Can You Do?
Identity is the new perimeter. In the old days of security, you protected the network edge — if you were inside the building you were trusted. That model is dead. Today, attackers get inside (via phishing, credential theft, stolen tokens) and then move laterally using legitimate-looking identities. The attacker isn't breaking down the door — they're walking in with a stolen badge.

IAM (Identity and Access Management) is the discipline of controlling who can access what, with how much permission, for how long. The guiding principle is Least Privilege — every person and every application gets the minimum access needed to do their job, nothing more.

Conditional Access is how you enforce that. Instead of just asking "do you have a password?" it asks: "Are you who you say you are? Are you on a trusted device? With PIM, admin access is off by default and turned on only when needed, for a limited time, with a business justification logged. This dramatically shrinks the window an attacker has to exploit a compromised admin account.

SOAR (Security Orchestration, Automation and Response) is using automation to respond to threats faster than any human can. Instead of a human seeing an alert, opening a ticket, and manually disabling an account — a Logic App does it in seconds, automatically, at 3am when no analyst is watching.

💡 Plain English — What This Actually Means

The most powerful security work happens when you correlate data across tools. Palo Alto log = this IP connected to this destination, firewall allowed it. Falcon log = this process made a suspicious network connection. Together: the firewall allowed it AND the endpoint knew something was wrong. The cross-tool correlation catches sophisticated attacks that evade individual tools. Building this correlation — pulling from both Palo Alto and Falcon into Splunk — is an advanced skill that very few candidates can demonstrate. It's your differentiator.
